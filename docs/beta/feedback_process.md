# FiduScan Feedback & Bug Reporting Process

## Overview
To ensure the beta phase is productive and actionable, we employ a structured triage system for all incoming tester feedback. This document outlines the expected workflows for reporting bugs and requesting features.

## 1. Submission Channels
- **Priority Triage (Bugs & Outages):** Use the private Slack channel `#fiduscan-beta-squad`. Engineering leads monitor this channel 24/7 during the beta window.
- **Structured Feedback (General UX & Feature Requests):** Submit the `feedback_template.md` via email to `beta-support@fiduscan.com`.

## 2. Bug Reporting Workflow
When you encounter a technical failure, please gather the following before reporting:
1. **The Request ID:** Visible on the Result Card UI. This UUID allows us to trace the exact server logs and inference times.
2. **The File Payload:** If non-confidential, please share the specific image, audio, or video file that triggered the crash. If it is confidential, describe the encoding format and size.
3. **The Environment:** Specify if you were testing via the Web UI or via the REST API.

*Example Slack message:*
> "Bug Report: Uploading a 45MB MKV video via the Web UI throws a 500 Internal Server Error after 3 seconds. Request ID: 8f72c-1293a. Tested on Safari 17."

## 3. False Positive / False Negative Reporting
If the AI detection engine misclassifies a file:
1. Do NOT delete the file from your Scan History.
2. Submit a report detailing the True Authenticity (e.g., "This image was generated by Midjourney v6").
3. We will correlate the cryptographic hash from your History with our model weights to isolate the failure point and fine-tune the ensemble.

## 4. Feature Request Workflow
If you identify a missing capability (e.g., "I need PDF reporting exports"):
1. Post a brief summary in the `#fiduscan-beta-features` Slack channel.
2. The product team will formalize this into a Jira ticket (`BETA-FR-XXXX`).
3. We encourage other testers to react with a 👍 to upvote features. High-priority features may be fast-tracked into the Phase 9 Production Live release.

## 5. Triage SLAs
- **Severity 1 (Critical Outage):** Investigated within 1 hour.
- **Severity 2 (Broken Core Feature):** Investigated within 4 hours.
- **Severity 3 (UX Glitch / Model Inaccuracy):** Logged for the next patch cycle.
