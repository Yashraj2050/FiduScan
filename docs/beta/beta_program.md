# FiduScan Beta Testing Program

## 1. Beta Program Goals
The primary objective of the Phase 8C Beta Program is to validate the FiduScan platform under real-world conditions before public commercial availability. Specifically, we aim to:
- Stress-test the unified multimodal engine (Image, Audio, Video).
- Evaluate UX clarity and dashboard performance under heavy payload conditions.
- Ensure Stripe test mode integrations correctly throttle and limit usage as intended.
- Validate our inference latency SLAs (Target: <200ms for images, <1.5s for video).

## 2. Target Tester Profiles
We are selecting testers who represent our ideal commercial user base:
1. **Trust & Safety Engineers:** Moderation team leads looking to filter synthetic media.
2. **Journalists/OSINT Analysts:** Verification experts looking for forensic explainability.
3. **Developers:** Engineers testing the resilience and ergonomics of our REST API.

## 3. Tester Count
- **Initial Goal:** 10–20 active users.

## 4. Feedback Collection Process
Feedback will be collected synchronously and asynchronously:
- **Asynchronous:** Via the FiduScan Feedback Template submitted to `beta-support@fiduscan.com`.
- **Synchronous:** A private Slack channel (`#fiduscan-beta-squad`) for real-time triage.
- **Automated:** We will monitor server logs and API telemetry to passively identify performance bottlenecks or backend 500 errors.

## 5. Bug Reporting Process
Users experiencing technical faults should submit a Bug Report containing:
1. Expected vs Actual Behavior.
2. The specific file type, size, and modality being tested.
3. Relevant request IDs generated by the UI.
*For detailed instructions, refer to `feedback_process.md`.*

## 6. Feature Request Process
Feature requests should focus on UX improvements, API endpoint modifications, or new modality support. These will be tracked in our public Jira board under the `BETA-FR` epic, allowing testers to upvote popular requests.

## 7. User Survey Template
Upon completion of the 14-day beta window, all testers will receive a mandatory exit survey.
*See `user_survey.md` for the complete questionnaire.*

## 8. Success Metrics
The beta program will be considered successful if we hit the following KPIs:
1. **Zero Critical Outages:** 100% uptime on the primary inference engine during the beta window.
2. **High Inference Accuracy:** < 2% reported false positive rate on the `AUTHENTIC` class.
3. **Rapid Triage:** Mean Time To Resolve (MTTR) for critical bugs < 24 hours.
4. **UX Score:** > 8.5/10 aggregate score on the exit survey.

## 9. Retention Metrics
- **Daily Active Users (DAU):** Target of 50% of the cohort logging in daily.
- **Scan Volume:** Target of > 500 cumulative scans across all modalities per user by the end of the beta.

## 10. Exit Criteria
We will transition out of Phase 8C and into Phase 9 (Production Live) when:
- 10+ testers have successfully completed the User Survey.
- 100% of reported `Severity 1` (Critical) and `Severity 2` (High) bugs are resolved.
- The Billing system correctly throttles users without failing open or closed erroneously.
